# Heartwith Protocol Design

## 心率数据特点

- 低频：小米手环标准心率通知通常是秒级或更低频率，不需要毫秒级实时链路。
- 小包：单个样本只需要时间偏移和 BPM，适合二进制批量编码。
- 容忍轻微延迟：共享大厅更关注最近状态和趋势，`5-15s` 延迟可接受。
- 单向为主：采集端只需要上传心率，服务端首版不需要频繁下发控制指令。
- 弱网常见：移动端应允许短时离线缓存、恢复后补发，避免每秒唤醒网络。

## 协议选择

移动采集端到服务端使用 `HTTPS POST` 批量上传，payload 首选 `application/cbor`。

不把 MQTT 或 WebSocket 长连接作为默认采集协议，原因是：

- 心率数据小且低频，长连接 keepalive 的收益低。
- 移动端后台和弱网下，长连接更容易触发额外唤醒和重连。
- HTTP/2/3 连接复用交给系统网络栈处理，更容易和系统调度合并。
- 服务端无需实时向采集端下发消息，单向批量上传足够。

网页大厅使用 SSE：

- 浏览器只接收服务端聚合后的大厅状态，不直接连接采集端。
- SSE 对大厅单向推送足够简单，首版不需要 WebSocket 的双向复杂度。

## 移动端上传策略

- BLE 侧只扫描绑定/重连，连接后订阅标准心率特征 `0x2A37`。
- 正常批量窗口：`8s`。
- 正常最大窗口：`8s`。
- 低电量窗口：`8s`。后台省电主要依靠不扫描、少更新通知、直连上次设备，而不是拉长上传延迟。
- 心率相对上次成功上传变化 `>= 3 bpm` 时提前 flush。
- 离线缓存最近 `5 分钟`。
- 缓存样本过多时，恢复上传按约 `5s` 粒度降采样，并保留最新样本。

## 幂等与时间

- `collector_id + seq` 是批量包幂等键，重复包返回 `accepted: 0`。
- `sent_at_ms` 是发送时间，样本使用 `dt_ms` 相对发送时间表达。
- 服务端在客户端时钟偏差超过 `60s` 时使用接收时间校正。
- 乱序包可以写入曲线，但不会覆盖更新的 `last_bpm`。
